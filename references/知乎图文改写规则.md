# 知乎图文改写规则

目标不是“把视频字幕排版一下”，而是“把适合口播的表达，重构成适合知乎阅读和收藏的图文稿”。

## 改写目标

生成的内容应该：

- 保留原作者的真实经验感
- 明显强于原始逐字稿的结构化程度
- 让没有看过视频的知乎读者也能独立读懂
- 可以继续人工打磨后直接发布
- 能把视频里零散提到的高价值信息，整理成更直观、更方便复制使用的 Markdown 结构

## 两阶段产物

知乎图文不要默认一步到位。

更合理的方式是分成两阶段：

### 1. 工作稿

用途：

- 确认主轴
- 确认重点
- 暴露缺失信息
- 先完成结构和正文骨架

工作稿允许保留轻度编辑痕迹，但不应在明知缺关键信息的情况下，把大量模糊表述和过程性备注直接写进正文。

### 2. 最终发布稿

用途：

- 真正用于发到知乎

最终发布稿不应保留：

- “待确认”
- “后续补充”
- “视频里没有明确提到”
- “视频里提到”“按视频里的演示”“根据视频里的描述”这类以源视频为前提的叙述视角
- 任何会让读者觉得这是内部过程稿的措辞

如果一篇稿子还存在这类内容，就说明它仍然只是工作稿。

最终发布稿的默认叙述视角，应是直接面向知乎读者的成品表达。

- 优先写成“这类 Skill 常见的入口是……”
- 不要写成“视频里提到的入口是……”
- 优先写成“更省事的一种做法是……”
- 不要写成“按视频里的演示……”

只有当文章本身就是在分析视频表达、拆解视频内容结构时，才允许保留“视频中/视频里”的表述。

在输出最终发布稿前，优先使用 [最终发布前检查清单](最终发布前检查清单.md) 逐项检查。

**图片嵌入格式**：最终版中嵌入截图或配图时，统一使用 `![[文件名.jpg]]`（Obsidian wikilink 语法），不要使用 `![alt](路径)` 语法。后者在文件名含空格或特殊字符时无法正确解析。

## 改写前先做内容缺口扫描

不要默认“逐字稿一到手就直接写正文”。

先检查这些内容是否明确：

- 这条内容的主轴是什么
- 用户最想强调的重点是什么
- 关键事实信息是否完整
- 发布目标和文章风格是否明确
- 哪些信息可能识别错误或判断错误

如果这些信息不明确，而且会影响文章方向或准确性，应先给用户一份“最小必要提问清单”，再继续成稿。

分析与改写时，优先参考 [证据等级规则](证据等级规则.md)，不要把低确定性信息直接写成高确定性事实。

### 最小必要提问清单的要求

- 只问 2 到 5 个关键问题
- 只问会改变文章方向、重点排序、信息准确性的点
- 一次性覆盖主轴、重点、事实补全、发布目标与必要边界
- 不要问开放过头的问题

示例：

- 这条内容你最希望读者看完记住什么？
- 这篇更应该写成哪一类文章？
- 你最想强调的 1 到 3 个信息点是什么？
- 有没有哪些事实信息必须确认或补齐？

## 一次性收集，不做第二轮提问

如果预计最终发布会需要用户补充的信息已经暴露出来，就应在第一次提问里一起问完，而不是等工作稿写完后再补第二轮。

也就是说：

- 提问应尽量发生在“逐字稿 -> 成稿”之间
- 不要默认“先写工作稿，再补问题”
- 工作稿不应该成为暴露明显信息缺口的替代方案

## 证据等级与写法关系

写工作稿和最终稿时，不要只看“这个信息有没有”，还要看“它凭什么成立”。

- A/B 级：可以作为事实使用
- C 级：非关键事实可用，关键事实要谨慎
- D 级：只能保守表达，不能写死
- E 级：不应进入最终稿

关键事实优先指：

- 工具 / 产品名称
- 仓库链接
- 安装入口
- 命令
- 快捷键
- 平台名称

## 推荐结构

### 1. 标题候选

给出 3 个标题候选，类型可以混合：

- 经验分享型
- 方法总结型
- 观点判断型

标题不要写成小红书式夸张钩子，也不要太论文腔。

### 2. 开头

开头优先做两件事：

- 用一句话说清文章讨论的核心问题
- 快速说明为什么这件事值得聊

不要把大量铺垫、寒暄和“今天想聊聊”之类口语原样保留。

### 3. 主体

主体通常拆成 3 个部分最稳：

- 为什么我开始这样做
- 这样做带来了哪些实际收益
- 适用边界、问题和注意点

如果原视频更偏教程，也可以改成：

- 背景
- 方法
- 示例
- 注意事项

### 4. 结尾

结尾不要机械总结。优先用下面两种方式之一：

- 抛一个有讨论空间的判断
- 点出下一篇可展开的话题

## Markdown 可视化增强

目标不是做“花哨排版”，而是把读者真正会用到的信息，从正文里抽出来，变成一眼能扫到、顺手能复制的结构。

### 1. 优先抽取的信息类型

- 下载地址、项目地址、官方文档、仓库链接
- 安装命令、调用命令、配置片段、提示词片段
- 模型分工、工具对比、优缺点对照
- 操作步骤、工作流顺序、判断条件
- 发布前注意事项、限制条件、踩坑点

### 2. 推荐的表现形式

#### 链接 / 地址

- 能确认的网址，优先写成 Markdown 链接
- 如果用户更需要“复制后直接打开”，可在链接下方再补一个纯文本代码块

示例：

```markdown
项目地址：
[名称](https://example.com)

复制地址：
```text
https://example.com
```
```

#### 命令 / 安装方式

- 所有 shell 命令、安装命令、调用命令，优先用代码块
- 不要把命令写进长段落里
- 如果命令来自转写且不确定，明确标注“待人工确认”

示例：

```markdown
## 可直接复制的命令

```bash
npx @example/skill
```

> 注：该命令来自视频口播，发布前建议再核对一次包名。
```

#### 对比关系

- 横向比较内容优先用表格，例如模型职责、优缺点、价格感知、适用边界

示例：

```markdown
| 模型 | 更适合做什么 | 短板 |
| --- | --- | --- |
| 方案工具 | 读材料、补上下文、整理方向 | 可能更慢或更贵 |
| 执行工具 | 改代码、补细节、落方案 | 表达和总结可能较弱 |
| 视觉工具 | 出展示稿、做配图、补视觉方向 | 长上下文稳定性可能一般 |
```

#### 重点判断

- 值得读者记住的判断，用短句列表或加粗句呈现
- 不要把所有句子都加粗，只强调结论句

#### 步骤流程

- 有前后顺序的内容，优先写成编号列表
- 每一步尽量只保留一个动作或一个判断

### 3. 何时必须做信息提炼

如果视频中出现以下任一类内容，就不应只写成普通正文：

- 明确提到了“下载地址 / 项目地址 / 官方链接”
- 明确给出了“命令 / 安装方式 / 调用方式”
- 明确出现了“角色分工 / 优缺点对照”
- 明确有“步骤链路 / 处理流程”

### 4. 不确定信息处理

- 转写中出现明显误识别时，不要自行脑补完整命令或链接
- 可以先写“待人工确认的命令”或“待核对链接”
- 如果只能确认部分信息，就保留已确认部分，并说明不确定点
- 宁可少写，也不要把错误命令当成最终答案输出

## 语言风格

### 保留的部分

- 第一人称经验
- 真实判断
- 具体例子
- 个性化表达

### 需要削弱的部分

- 口头禅
- 重复解释
- 明显依赖语气词的停顿
- 纯情绪化夸张表达

## 风格偏向

默认不要写成“平台标准答案”。

更接近目标的风格是：

- 有第一人称判断
- 有复盘感
- 有轻微的个人情绪和现场感
- 有明确结论，但不是说教腔

如果能从用户现有内容里观察到稳定写法，优先贴近用户自己的表达，而不是贴近泛化的知乎大 V 写法。

尤其避免：

- “本文将”
- “我们不妨”
- “从三个维度来看”
- “在当下 AI 快速发展的背景下”
- 任何过于模板化的开头

## 内容重组原则

### 1. 不逐句改写

必须先提炼结构，再成文。

### 2. 先找核心结论

通常一条视频真正想表达的核心不超过 1 到 2 个。先找出来，再决定哪些句子该保留。

### 2.5 优先保留作者自己的判断句

如果原视频里有明显的“我后来越来越觉得”“真正重要的是”“我最在意的是”这类判断句，不要轻易删掉。

这类句子往往正是作者风格和文章灵魂所在。

### 3. 用“读者获得感”替代“视频节奏感”

视频里靠节奏推动的内容，图文里要换成：

- 小标题
- 结论先行
- 例子支撑
- 过渡句

### 4. 保留具体案例

如果原视频讲的是个人工作流、真实项目、工具协作，一定保留案例，不要改成空洞观点文。

## 默认输出格式

建议用以下结构输出知乎版初稿：

```markdown
# <原标题>｜知乎图文工作稿

## 标题候选
- ...
- ...
- ...

## 内容定位
- 目标平台：知乎
- 内容类型：经验分享 / 方法教程 / 观点分析 / 复盘
- 核心价值：...

## 快速结论
- ...
- ...

## 正文
...

## 关键信息提炼

### 可直接复制的命令
```bash
# 如果有
```

### 链接与资源
- [名称](https://example.com)

### 对照表 / 步骤表
| 项目 | 内容 |
| --- | --- |
| ... | ... |

## 可延展选题
- ...
```

在用户补充完成后，再生成一份：

```markdown
# <原标题>｜知乎图文最终版
```

## 质量门槛

提交前至少检查：

- 是否已经脱离“字幕感”
- 是否有稳定的小标题结构
- 是否保留了原作者独特判断
- 是否让知乎读者不看视频也能读懂
- 是否还有明显的小红书话术残留
- 是否把真正有复用价值的信息，从正文里提炼成了更直观的 Markdown 结构
- 如果这是最终发布稿，是否已经去掉所有过程性说明与模糊表述
