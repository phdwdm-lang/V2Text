# 高质量视频理解提示词

目标不是只把视频声音转成文字，而是同时理解：

- 这条视频在讲什么
- 这条视频在做什么操作
- 哪些事实信息可以直接复用到知乎图文里

## 默认提示词

```text
你现在要分析的是一条小红书视频。请不要只做逐字稿总结，而是把“视频里讲了什么”和“视频里做了什么操作”一起整理出来。

你的任务目标：
1. 准确总结这条视频的核心主题、目标和结论。
2. 尽可能记录视频中的具体操作过程，而不仅仅是口播内容。
3. 识别并提取视频中出现的工具、产品、网站、平台、模型、命令、路径、按钮、页面切换、代码操作。
4. 为后续改写成知乎图文内容提供可复用素材。

分析要求：
- 不要只复述字幕。
- 要结合画面、字幕、口播、页面变化一起理解。
- 如果画面中出现了操作演示，要记录“做了什么动作、在哪个页面、使用了什么工具、动作的目的是什么”。
- 如果视频中提到了链接、网站、产品名、命令、快捷键、仓库名、Skill 名称、模型名称等，请尽量提取。
- 如果某个专有名词听不清或看不清，不要瞎猜，标注为“待确认”，并说明你为什么不确定。
- 对关键事实尽量给出证据等级，优先区分：
  - 用户确认
  - 画面可确认
  - 口播可确认
  - 多模态推断
  - 待确认
- 如果视频是“功能更新介绍”或“工具演示”，请重点突出“更新了什么”“怎么操作”“适合谁用”。
- 如果视频是“经验分享”或“观点表达”，请重点突出“作者核心观点”“支撑案例”“操作只是辅助”。

输出格式请严格按下面结构：

# 视频分析摘要

## 1. 一句话总结
- 用 1 到 2 句话说明这条视频在讲什么、想传达什么。

## 2. 核心内容总结
- 用 3 到 6 条要点总结视频最重要的信息。
- 这些要点应适合后续改写成知乎正文的小标题或段落。

## 3. 视频操作时间线
请按时间顺序输出表格，字段如下：
- 时间段
- 当前画面 / 页面
- 用户做了什么操作
- 涉及的工具 / 网站 / 产品
- 这一步的目的
- 备注 / 待确认信息

如果某一段没有明显操作，可以写“口头说明为主”。

## 4. 关键信息提取
按列表提取：
- 产品 / 工具名称
- 网站 / 平台名称
- 代码相关操作
- 命令 / 快捷键
- 文件 / 路径 / 仓库 / Skill 名称
- 适用人群
- 作者给出的建议或结论

对关键事实请尽量附一项：
- 证据等级

## 5. 可直接复用的事实卡片
请整理成短条目，适合后续做 Markdown 表格、代码块、信息卡片：
- 名称：
- 类型：
- 作用：
- 如何使用：
- 注意事项：

## 6. 后续写成知乎图文时最值得强调的 3 个点
- 点 1
- 点 2
- 点 3

## 7. 不确定信息
- 列出所有你无法 100% 确认、但可能影响最终成稿准确性的内容。
- 每条后面注明：为什么不确定。

额外要求：
- 输出必须是中文。
- 表达要清晰、具体、结构化。
- 优先保留“能帮助读者理解和复现”的信息。
- 如果视频中既有讲解又有操作，请不要偏向其中一边，必须两者都覆盖。
```

## 产物建议

高质量视频理解模式下，默认产物建议为：

1. `视频分析稿.md`
2. `知乎图文工作稿.md`
3. `知乎图文最终版.md`

默认不再额外生成口播稿 / 清洗逐字稿，除非用户明确要求保留。

视频分析稿的落库结构，优先参考：

- [视频分析稿模板](视频分析稿模板.md)
- [证据等级规则](证据等级规则.md)

## 截图候选列表说明

脚本提示词中已包含截图候选列表的生成要求，模型输出应为如下格式的 Markdown 表格：

| 精确时间（秒） | 截图类型 | 用途说明 | 是否必须 |
| --- | --- | --- | --- |
| 32s | 操作步骤 | 说明截这帧的目的 | 必须 |

截图类型枚举：`操作步骤` / `界面全局` / `关键结果` / `前后对比` / `工具界面`

规则：
- 时间精确到秒，格式为 `32s`，不使用 `00:32` 格式
- 操作演示型视频应填写 5 行以上
- "是否必须"仅填 `必须` 或 `可选`，无强需求的截图不填
- 不要先截图再倒推说明，候选列表由视频理解模型直接给出
