# 图片生成规则

本文件聚焦**生成执行**：在已经完成图类型判断（参考[可视化选型与出图流程](可视化选型与出图流程.md)）和视觉规范确认（参考[知乎配图视觉规范](知乎配图视觉规范.md)）后，如何将内容转化为可交付的 HTML/SVG 代码。

---

## 前置条件

进入生成流程前，必须满足：

1. 知乎图文**最终版**已完成并通过发布前检查
2. 已按[可视化选型与出图流程](可视化选型与出图流程.md)确认：需要出图、图类型已定
3. 已明确这张图在正文中的插入位置

**工作稿阶段不生成正式配图**，可以写"可视化建议"占位。

---

## 第一步：生成视觉文案

在写任何代码之前，先输出一份极短的视觉文案：

```
设计方向（内部锚点，不出现在图上）：这张图让读者理解 ___
图类型：
主标题：（图内展示，20 字以内）
副标题：（可选，1 行以内）
关键标注：（2 到 5 条，每条不超过 20 字）
底部收口：（可选，主体足够时留空）
```

**关于设计方向**：设计方向决定布局、数据取舍和视觉重心，但**禁止直接复制为图内标签、标题或标语**。图上展示的标题和标注必须用具体的"图用语言"重新提炼。

**判断标准**：如果设计方向写不出来，说明还不该出图；如果写出来的视觉文案仍然像正文段落的压缩版，说明图的信息层还没有和正文分离，不能继续写代码。

---

## 第二步：生成 HTML/SVG 代码

### 通用约束

- 一张图只表达一个信息中心
- 图内文字使用"图用语言"：短标题 + 关键标注 + 可选收口，不搬运正文段落
- 主体内容必须放在有边界的视觉容器（卡片、分组框）内，不浮在背景上
- 标题区不堆砌多余元素（标签、摘要条），挤压主体空间
- 底部有空白时补收口标签，主体已饱满时不加

### 对比图（HTML/CSS）

- 左右双栏布局，每栏信息量对等
- 栏间用视觉分隔线或对比箭头明确"vs"关系
- 每栏 2 到 4 个要点，不要把整段正文塞进一栏

### 角色分工图（HTML/CSS）

- 2 到 4 栏卡片，每张卡片对应一个角色/模块
- 卡片内：角色名（大字）+ 核心职责（2 到 3 条）+ 可选短板
- 不同角色用不同主色区分，但整图主色系保持一致

### 流程图（Mermaid + HTML 容器）

- Mermaid 负责节点和逻辑关系
- 外层 HTML 容器提供：标题区、背景色、内边距
- Mermaid 不裸输出，必须包裹在 HTML 容器内
- 节点文字简短（5 到 10 字），不要把说明段落写进节点

### 步骤图（HTML/CSS）

- 纵向编号列表，每步 1 个动作或 1 个判断
- 步骤之间用连接线或箭头表达顺序
- 每步文字控制在 2 行以内

### 总结图（HTML/CSS）

- 3 到 4 张卡片横排或竖排
- 每张卡片：标题（核心词）+ 1 到 2 句说明
- 不要在每张卡片里写完整段落

### 数据图（HTML/CSS）

- 优先用百分比条、对比数值卡、简单柱状图
- 数据来源必须是正文中已确认的信息（A/B 级证据）
- 不捏造或估算数字

## 第三步：HTML 导出为 PNG

生成的 HTML 信息图必须导出为 PNG 图片后才能嵌入最终版。

### 导出方案：Puppeteer 截图

```javascript
// screenshot.js — 放在配图目录下
const puppeteer = require('puppeteer');
const path = require('path');

(async () => {
  const browser = await puppeteer.launch();
  const page = await browser.newPage();
  // viewport 宽度必须大于内容宽度，避免水平滚动条
  await page.setViewport({ width: 1200, height: 2000, deviceScaleFactor: 1 });

  const htmlFile = path.resolve(__dirname, '目标文件.html');
  await page.goto('file:///' + htmlFile.replace(/\\/g, '/'), { waitUntil: 'networkidle0' });

  const height = await page.evaluate(() => document.body.offsetHeight);

  await page.screenshot({
    path: path.resolve(__dirname, '../截图/输出文件名.png'),
    clip: { x: 0, y: 0, width: 1080, height: height }
  });

  await browser.close();
})();
```

### 关键参数

| 参数 | 值 | 原因 |
|---|---|---|
| viewport 宽度 | `1200`（大于内容 1080） | 避免水平滚动条导致高度计算错误 |
| viewport 高度 | `2000`（足够大） | 让页面完整渲染 |
| 裁切方式 | `clip` 参数 | 精确裁切到 `body.offsetHeight`，不依赖 viewport |
| 内容宽度 | `1080` | 知乎配图标准宽度 |

### HTML 侧配合

body 需设置 `display: inline-block`，确保 `offsetHeight` 精确反映内容高度：

```css
html, body { margin: 0; padding: 0; }
body { width: 1080px; display: inline-block; }
```

### 导出后检查

- 底部无大面积空白（padding-bottom 控制在 16-24px）
- 无水平滚动条残留
- 用图片查看器打开确认尺寸，不要仅靠 Markdown 预览器判断（Obsidian 等预览器可能在图片下方添加额外容器空间）

## 第四步：截图处理（如需）

本步骤只处理 SKILL.md 步骤 8（截图适配性验证）中判定为**「裁切后可用」**的截图。判定为「适合」的截图直接使用，判定为「不适合」的截图已在步骤 8 中被替换或标记为重绘。

截图只作为辅助证据，不作为配图主体。

处理前确认：
1. 关键区域是否足够大且清晰
2. 手机上看关键文字是否可读
3. 需要裁切掉哪些无关内容

处理顺序：**裁切 → 放大 → 添加标注框 → 仍不清晰则改为重绘信息图**

代码类截图（代码、控制台、文件树）不要让用户在配图里直接阅读细字，优先提炼成信息卡片，截图只做背景佐证。

---

## 第五步：出图后自检

回答以下 5 个问题，有 2 条以上答不出则重新设计：

1. 这张图帮读者理解了正文中的哪个难点？
2. 图内文字比正文更短更直接了吗？
3. 底部是否没有大面积空白？
4. 手机宽度下标题和标注是否仍可读？
5. 读者能在 3 秒内知道这张图要表达什么吗？

---

## 第六步：导出前执行检查清单

执行 [配图检查清单](配图检查清单.md) 全部检查项，未通过不导出。
